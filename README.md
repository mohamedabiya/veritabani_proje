1. PROJE SENARYOSU
SENARYO
Günümüzde e-ticaret sistemleri, kullanıcıların internet üzerinden ürünleri inceleyip satın almasına sağlayan önemli platformlardır. Bu sistemlerin sağlıklı ve verimli çalışabilmesi için güçlü bir veritabanı tasarımına ihtiyaç duyulmaktadır. Bu çalışmada, bir e-ticaret uygulaması için ilişkisel veritabanı tasarımı ve senaryosu ele alınmaktadır.

Sistemde iki tür kullanıcı bulunmaktadır: müşteri ve satıcı. Satıcılar kendi mağazalarını oluşturabilir ve ürün ekleme, güncelleme ve stok yönetimi işlemlerini gerçekleştirebilirler.
Her ürün yalnızca bir kategoriye ait olacaktır. Bir kategori içerisinde birden fazla ürün bulunabilir. Ürünlere ait görseller sistemde saklanacak ve her ürün için birden fazla resim eklenebilecektir.
Müşteriler satın almak istedikleri ürünleri sepete ekleyebilir, sepet içerisindeki ürünlerin miktarını değiştirebilir veya ürünleri sepetten kaldırabilirler. Her müşterinin yalnızca bir aktif sepeti bulunacaktır.
Sipariş oluşturulduğunda, sipariş içerisindeki her ürün için ürün miktarı ve birim fiyat bilgileri ayrıca saklanacaktır. Böylece geçmiş siparişlerde ürün fiyatları değişse bile eski sipariş bilgileri korunabilecektir.
Sistem, farklı ödeme yöntemlerini desteklemektedir. Müşteriler kredi kartı ile ödeme veya kapıda ödeme seçeneklerinden birini tercih edebilirler. Kredi kartı ödemelerinde kart sahibi adı, kart numarası, son kullanma tarihi ve güvenlik kodu bilgileri doğrulama amacıyla kullanılacaktır. Kapıda ödeme yönteminde ise teslimat sırasında doğrulama kodu kullanılacaktır.
Sipariş durumları “Hazırlanıyor”, “Kargoda”, “Teslim Edildi” ve “İptal Edildi” gibi değerler alabilecektir. Sistem sipariş durum değişikliklerini takip ederek müşteriye bilgi sağlayacaktır.
Bir müşteri geçmiş siparişlerini görüntüleyebilir ve hangi ürünleri ne zaman satın aldığını inceleyebilir. Ayrıca sistem, ürün stok miktarını otomatik olarak güncelleyecektir. Sipariş tamamlandığında ilgili ürünlerin stok miktarı azaltılacaktır.
Veritabanı tasarımında veri bütünlüğünü sağlamak amacıyla birincil anahtar (Primary Key) ve yabancı anahtar (Foreign Key) ilişkileri kullanılacaktır. Böylece kullanıcılar, siparişler, ürünler, kategoriler, ödemeler ve adresler arasındaki ilişkiler güvenli şekilde yönetilecektir.

5. SQL DDL İFADELERİ
5.1 Tablo Oluşturma (CREATE TABLE)
create table app_user (
   id            number generated by default as identity primary key,
   first_name    varchar2(100) not null,
   last_name     varchar2(100) not null,
   email         varchar2(150) not null unique,
   password_hash varchar2(255) not null,
   phone_number  varchar2(20) not null unique
);

create table customer (
   id number primary key,
   constraint fk_customer_user foreign key ( id )
      references app_user ( id )
         on delete cascade
);

create table sales (
   id         number primary key,
   store_name varchar2(150) not null,
   constraint fk_sales_user foreign key ( id )
      references app_user ( id )
         on delete cascade
);




create table address (
   id           number generated by default as identity primary key,
   user_id      number not null,
   full_address clob not null,
   constraint fk_address_user foreign key ( user_id )
      references app_user ( id )
         on delete cascade
);

create table category (
   id   number generated by default as identity primary key,
   name varchar2(100) not null
);

create table product (
   id          number generated by default as identity primary key,
   user_id     number not null,
   category_id number not null,
   name        varchar2(200) not null,
   price       number(10,2) not null,
   stock       number default 0,
   constraint fk_product_user foreign key ( user_id )
      references sales ( id )
         on delete cascade,
   constraint fk_product_category foreign key ( category_id )
      references category ( id )
);

create table picture (
   id         number generated by default as identity primary key,
   product_id number not null,
   image_path varchar2(500) not null,
   constraint fk_picture_product foreign key ( product_id )
      references product ( id )
         on delete cascade
);






create table orders (
   id           number generated by default as identity primary key,
   user_id      number not null,
   address_id   number not null,
   total_price  number(10,2) not null,
   order_status varchar2(20) default 'pending',
   order_date   timestamp default current_timestamp,
   constraint chk_order_status
      check ( order_status in ( 'pending',
                                'processing',
                                'shipped',
                                'delivered',
                                'cancelled' ) ),
   constraint fk_orders_user foreign key ( user_id )
      references customer ( id ),
   constraint fk_orders_address foreign key ( address_id )
      references address ( id )
);

create table order_item (
   id         number generated by default as identity primary key,
   order_id   number not null,
   product_id number not null,
   price      number(10,2) not null,
   quantity   number default 1,
   constraint fk_orderitem_order foreign key ( order_id )
      references orders ( id )
         on delete cascade,
   constraint fk_orderitem_product foreign key ( product_id )
      references product ( id )
);

create table payment (
   id             number generated by default as identity primary key,
   order_id       number not null unique,
   payment_method varchar2(50) not null,
   payment_status varchar2(50) default 'pending',
   constraint fk_payment_order foreign key ( order_id )
      references orders ( id )
         on delete cascade
);


create table payment_by_card (
   id          number primary key,
   card_number varchar2(20) not null,
   cvv         varchar2(4) not null,
   end_date    date not null,
   constraint fk_card_payment foreign key ( id )
      references payment ( id )
         on delete cascade
);


create table cash_on_delivery (
   id          number primary key,
   verify_code varchar2(20),
   constraint fk_cash_payment foreign key ( id )
      references payment ( id )
         on delete cascade
);

create table cart (
   id      number generated by default as identity primary key,
   user_id number not null unique,
   constraint fk_cart_user foreign key ( user_id )
      references customer ( id )
         on delete cascade
);

create table cart_item (
   id         number generated by default as identity primary key,
   cart_id    number not null,
   product_id number not null,
   quantity   number default 1,
   constraint fk_cartitem_cart foreign key ( cart_id )
      references cart ( id )
         on delete cascade,
   constraint fk_cartitem_product foreign key ( product_id )
      references product ( id )
         on delete cascade
);

8. SONUÇ
Bu proje kapsamında, modern e-ticaret gereksinimlerini karşılayan, yüksek performanslı ve güvenli bir Sipariş Yönetim Sistemi Veritabanı başarıyla tasarlanmış ve Oracle SQL ortamında uygulanmıştır. Proje süreci boyunca şu kritik adımlar ve teknik başarılar gerçekleştirilmiştir:

●	İlişkisel Mimari ve Kalıtım Modellemesi: Kullanıcı rolleri(Satıcı,Müşteri) ve ve ödeme yöntemleri(Kapıda ödeme ve kart ile ödeme) için kalıtım modelleri tasarlanarak merkezi veri yönetimi ve genişletilebilirlik sağlanmıştır.

●	Veri Bütünlüğü ve İlişkiler: PRİMERY KEY ve FOREİGN KEY kısıtları eksiksiz uygulanmış.

Sistem, gerçek bir e-ticaret platformunun dinamik operasyonel ihtiyaçlarını ve yüksek hacimli veri akışını sorunsuz şekilde karşılayacak düzeyde normalize edilmiş (3NF) ve bütünlük kısıtları eksiksiz biçimde tanımlanarak proje senaryosuna uygun olarak başarıyla tamamlanmıştır.
